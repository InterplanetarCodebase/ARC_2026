## 3. Work Performed
Implementations and updates from today.

### 3.1 ODrive Calibration Automation (`odrive_calibration.py`)
- Developed automated script to configure and calibrate all **4 ODrives sequentially** in a single execution.
- **Key Parameters:**
  - **Serial numbers:** Hardcoded for reproducibility and safety
  - **Control mode:** Velocity control (default for rover differential drive)
  - **Velocity limit:** 10 turns/s (**PROHIBITED** to exceed, firmware-limited for safety)
  - **Motor current limit:** `axis.motor.config.current_lim = 20.0 A` (**CRITICAL FINDING**)

- **Critical Motor Current Discovery:**
  - Initial value of 10.0 A insufficient for obstacle traverse
  - At 10.0 A: wheels cannot overcome resistance, become trapped
  - At 20.0 A: wheels provide adequate torque to push through obstacles
  - This parameter is **non-negotiable** for field operation
  - Must be set during every calibration run

- **Calibration Workflow:**
  1. Erase existing configuration
  2. Apply motor/encoder settings (15 pole pairs, 90 CPR)
  3. Run full calibration (~40s per axis)
  4. Enable closed-loop velocity control
  
- **Execution Environment:** Runs on any device with direct ODrive USB connection (workstation, base station, or Jetson)

### 3.2 Differential Drive Odometry Node (`wheel_odom_node.py`)
- Standalone ROS2 node for converting wheel encoder readings to pose estimate + standardized odometry messages.
- **Key Measurements & Calibration:**
  - **Z-axis rotation scale factor:** 0.5 (**EMPIRICALLY TUNED**)
    - Without scaling: rotation appears 2× larger than ZED ground truth
    - Root cause: Differential drive kinematics formula calculates full rotation change
    - Solution: Multiply angular velocity by 0.5 before integration
    - Calibration method: Synchronized collection of wheel + ZED pose, statistical analysis of rotation bias
  
  - **Right-wheel inversion:** FR and RR motors physically wired inverted
    - Compensation: Negate encoder values for right-side wheels during pose calculation
    - Enables symmetric differential drive forward/backward response
  
- **Published Topics:**
  - `/wheel_odom` (nav_msgs/Odometry) — odometry frame with velocity twist
  - `/wheel_pose` (geometry_msgs/PoseStamped) — current 6-DOF pose only
  - `/tf` broadcast: `odom → base_link`

- **Execution Environment:** Base station (ROS2 node)

- **Why this script:** Decoupled odometry calculation allows for isolated testing and tuning before integration into drive controller.

### 3.3 Data Acquisition for Calibration (`collect_wheel_zed_raw.py` / Pose GUI)
- Two complementary data collection workflows:
  1. **CSV-based synchronized logging** (`collect_wheel_zed_raw.py`):
     - Subscribes to `/wheel_pose` and `/zed/zed_node/pose` (PoseStamped, not Odometry)
     - Records encoder-based rotation vs. ZED-based rotation with timestamps
     - Key CSV columns: `yaw_error_rad`, `d_wheel_yaw_rad`, `d_zed_yaw_rad`, `cum_wheel_yaw_rad`, `cum_zed_yaw_rad`, `yaw_gain_est`
     - Output: Enables statistical analysis of rotation deviation

  2. **Interactive PyQt5 GUI** (`pose_data_acquisition_gui.py`):
     - Real-time 4-plot display: ZED position, ZED orientation, wheel position, wheel orientation
     - **Features:**
       - Live streaming from ROS2 pose topics
       - JSONL recording with background async writer thread (prevents UI freeze at 100 Hz)
       - Session replay with speed control (0.25× to 4×)
       - Timeline slider for manual scrubbing
     - **Why JSONL format:** Human-readable JSON per line; enables easy filtering and post-processing
     - **Why background writer:** Synchronous file I/O at 100 Hz blocks UI thread; async queue-based buffering solved this

- **Execution Environment:** Base station (ROS2 subscriber)

- **Key Finding:** Analysis of collected data revealed **2× rotation overshoot** in wheel odometry vs. ZED ground truth. This drove the 0.5 scale factor.

### 3.4 Integrated WebSocket + ROS2 Controller on Jetson (`ws_to_diffdrive_ros.py`)
- **Primary production script for Jetson Xavier.**
- **Architecture:**
  - WebSocket server listens on `ws://0.0.0.0:8765` for drive commands from base station
  - Decodes differential drive command (forward/turn) to left/right wheel velocities
  - Commands ODrive motors with velocity setpoint
  - Publishes motor feedback as ROS2 topics visible on base station
  - Maintains background telemetry task at configurable rate (default 100 Hz)

- **Critical Parameter: Velocity Ramp Rate**
  - `vel_ramp_rate = 20.0` turns/s² (set in motor config)
  - Controls acceleration/deceleration slope
  - Higher ramp rate = faster response to velocity commands
  - **Purpose:** Ensures rapid motor stop when GCS signal halts; improves safety and responsiveness
  - Telemetry latency on WebSocket ≈ 50–200 ms; fast ramp compensates for this delay

- **ROS2 Topics Published:**
  - `/wheel_odom` (nav_msgs/Odometry)
  - `/wheel_pose` (geometry_msgs/PoseStamped)
  - `/tf` broadcast: `odom → base_link`

- **Command-Line Arguments:**
  - `--max_vel 4.0` — Maximum wheel speed (1.0–10.0 turns/s)
  - `--yaw_scale 0.5` — Calibration factor for Z-axis rotation
  - `--odom_rate_hz 100.0` — Odometry publish rate (Hz)

- **Execution Environment:** Jetson Xavier (connected via ethernet to base station)

- **Why this script:** Unifies motor control and odometry publishing in one process; eliminates need for separate `wheel_odom_node.py` on Jetson.

### 3.5 ROS-Free Alternative Controller (`ws_to_diffdrive_rosless.py`)
- **Identical to `ws_to_diffdrive_ros.py` but without ROS2 dependencies.**
- **Use case:** When Jetson and base station have incompatible ROS distributions or when minimal footprint is required.
- **Behavior:**
  - Still listens to WebSocket and commands ODrives
  - Does NOT publish ROS topics
  - Returns raw telemetry JSON over WebSocket only
- **Execution Environment:** Jetson Xavier (when ROS unavailable)

- **Why two variants:** ROS2 compatibility across devices is common blocker; rosless variant ensures deployment flexibility.

### 3.6 BLDC Telemetry GUI (`bldc_data_acquisition_gui.py`)
- **PyQt5 GUI for live motor telemetry visualization over WebSocket.**
- **Displays:**
  - Real-time motor currents (Iq, Id, Ibus) for all 4 motors (plots per ODrive board)
  - Motor temperature (FET thermistor)
  - Error states (axis, motor, encoder, controller errors as hex codes)
  - Bus voltage (Vbus0, Vbus1)
  - Sequence number (packet counter for diagnostics)

- **Features:**
  - Live 2D plotting (PyQtGraph, OpenGL fallback)
  - 3D OpenGL pose visualization with trail history
  - JSONL recording of full motor state snapshots
  - Replay/timeline scrubbing with speed control
  - Color-coded per-motor displays

- **Execution Environment:** Base station (WebSocket client)

- **Purpose:** Offline troubleshooting, motor health monitoring, telemetry diagnostics during field tests.

### 3.7 Motor Configuration & Parameter Justification
- **Pole Pairs:** 15 (hardware fixed)
- **Hall Encoder:** 90 CPR (hardware fixed)
- **Velocity Control Mode:** Enables smooth ramp-based deceleration; safer than position control
- **Velocity Limit:** 10 turns/s (firmware hard limit for safety)
- **Current Limit:** 20.0 A (motor thermal + obstacle handling balance)
- **Velocity Ramp Rate:** 20.0 turns/s² (rapid response to GCS stop commands)
- **Yaw Scale Calibration:** 0.5 (empirical vs. ZED ground truth)

## 7. Key Learnings & Discoveries

### 7.1 Motor Current Affects Obstacle Handling
- Systematic testing revealed 10.0 A insufficient for field operation
- Real-world obstacles (grass, rocks, slight inclines) require minimum 20.0 A for smooth traverse
- This is now hardcoded as non-negotiable in calibration script

### 7.2 Yaw Rotation Requires Empirical Calibration
- Differential drive formula does not inherently match hardware kinematics
- ZED ground truth revealed 2× overshoot in wheel-based rotation estimates
- Solution: scale factor applied during integration, not raw measurement
- Demonstrates importance of ground-truth sensor for validation

### 7.3 Telemetry Over WebSocket Adds Latency
- Typical round-trip: 50–200 ms from Jetson to base station
- Mitigation: high velocity ramp rate (20.0 turns/s²) ensures rapid motor response independent of telemetry feedback loop
- Safety mechanism: GCS halt command quickly decelerated via motor controller settings

### 7.4 Background Threading for UI Responsiveness
- Initial pose_data_acquisition_gui.py froze at 100 Hz due to synchronous file I/O
- Solution: async queue + background writer thread prevents UI thread blocking
- General principle: never perform blocking I/O on publisher/UI thread in high-rate streaming applications

### 7.5 Architecture Flexibility via Variants
- ROS2 vs. rosless split enables deployment flexibility
- Jetson + base station often have different OS versions or ROS versions installed
- Maintaining both variants (integrated and minimal) avoids re-architecture during deployment crisis
